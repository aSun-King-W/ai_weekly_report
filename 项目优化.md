# AI 周报项目优化方案

> 目标：打造一个有数据、可迭代、能写进简历的 AI Agent 项目

---

## Layer 1：修复评估系统（地基）

### 现状
- 已有 10 个测试用例，但真实环境下评估 0/10 通过
- 评价标准仅靠关键词匹配 + 简单结构检查
- `agent-evaluation-report.md` 显示大部分用例返回仅 20 字符

### 改进方案

#### 1.1 修复真实 API 调用链路
- [agent.test.ts](src/tests/agent.test.ts) `realExecuteQuery` 中配置的是 `ANTHROPIC_API_KEY`，但实际使用 DeepSeek API，确认环境变量是否正确
- [agent-service.ts](src/lib/agent-service.ts) 的 `processQuery` 调用链：确认 tool calling 的响应解析逻辑，确保在各种情况下都能返回合理的 `content`
- 添加详细的错误分类日志，区分「API 调用失败」「tool 执行失败」「响应解析失败」

#### 1.2 引入 LLM-as-Judge 评估
替代当前的关键词匹配，让 AI 模型自己判断输出质量：

```typescript
// 新增：AIJudge 评估器
interface JudgeCriteria {
  relevance: number;    // 内容相关性 0-10
  completeness: number; // 覆盖度 0-10  
  accuracy: number;     // 准确性 0-10
  structure: number;    // 格式规范 0-10
}
```

评估流程：给 DeepSeek 发送 `{query, output, criteria}` → 返回量化评分 + 评语。
- **文件**：新增 [src/utils/ai-judge.ts](src/utils/ai-judge.ts)
- 关键词匹配保留作为辅助参考，LLM-as-Judge 作为主要通过标准（综合分 >= 7/10 算通过）

#### 1.3 补充更多测试场景
现有 10 个用例已经不错，但可以补充：
- 英文 query 输入
- 超长仓库名
- 连续两次相同请求（缓存/幂等）
- 无网络情况下的优雅降级

#### 1.4 引入测试运行器
配置 Jest + ts-jest，让评估可以 `npm run test` 一键执行。
- 新增 `jest.config.ts`
- `package.json` 新增 `test`、`test:eval`、`test:eval:real` 脚本

#### 1.5 评估报告升级
当前 Markdown 报告缺乏可视化，改为输出 HTML 报告，包含：
- 通过率趋势图（对比历次运行）
- 各维度雷达图（相关性、覆盖度、准确性、结构）
- 失败用例的详细错误分析

## Layer 3：数据驱动迭代优化

### 核心循环
```
测量基线 → 分析瓶颈 → 实施改进 → 验证效果 → 重复
```

### 3.1 建立基线
运行 Layer 1 的评估系统，记录当前分数作为基线（比如准确率 80%）。
记录方式：保存每次评估结果到 `eval-results/` 目录，便于趋势对比。

### 3.2 分析失败模式
通过 Layer 2 的日志 + Layer 1 的详细报告，分类失败原因：
| 失败类型 | 可能原因 | 数据来源 |
|---------|---------|---------|
| tool 选择错误 | tool 描述不够精确 | agent-service 日志 |
| 输出结构混乱 | system prompt 缺少格式约束 | 评估报告结构检查 |
| 内容偏离 query | 上下文组装逻辑问题 | LLM-as-Judge 评语 |
| API 调用失败 | 网络/认证问题 | 错误日志 |

### 3.3 针对性优化

**优化点 A：Tool 描述优化**
- [tools.ts](src/lib/tools.ts) 中 tool 的 `description` 字段对 DeepSeek 的 function calling 准确性至关重要
- 分析哪些 query 选错了 tool，对应修改 tool 描述的措辞
- 示例改进方向：增加参数说明、添加使用场景示例

**优化点 B：System Prompt 优化**
- [agent-service.ts](src/lib/agent-service.ts) 中组装给 DeepSeek 的 system message
- 当前 prompt 可能存在指令模糊或缺少格式约束
- 改进方向：明确输出格式要求、添加 CoT（思维链）引导、示例 few-shot

**优化点 C：上下文组装优化**
- `processQuery` 中如何将 tool 返回的数据组装到下一轮对话
- 如果返回数据过大，需要做摘要/截断

**优化点 D：错误处理增强**
- 空仓库、无效仓库、API 限流等边缘情况返回友好的错误信息

### 3.4 每轮迭代模板
1. 在 `agent-evaluation-report.md` 记录当前分数
2. 实施 1-2 个优化点
3. 运行 `npm run test:eval:real` 验证
4. 记录新分数，对比前一轮
5. 如果达到目标（如 92%），记录最终报告

### 3.5 持续追踪
- 配置定时任务（如 GitHub Actions cron job），每周自动跑一轮评估
- 生成趋势报告，监控是否有回退

### 3.6 Layer 3 第2轮优化：消除内容虚构（已完成）

#### 现状（Iter 4，LLM-as-Judge 评估）
- **AI Judge 通过率**: 64.3%（9/14），平均分 7.79/10
- **关键词匹配通过率**: 27.1%（严重低估）
- **核心问题**: 4个失败用例中 3 个因模型**虚构提交数据**导致低分

#### 根因分析
当前架构存在 3 条幻觉路径：

```
用户查询 → Agent 模型 → get_github_commits → 真实 commits JSON → Agent 模型重新解析 → 模型自行构造 generate_report 参数 → aiService.generateReport(参数)
                                                          ↑                              ↑
                                                   路径3: 报告LLM在真实数据            路径2: 模型传入编造的commits
                                                   上额外虚构细节                      作为参数
                    
路径1: 模型跳过 generate_report 直接输出文本
```

**关键漏洞**：
- [agent-service.ts](src/lib/agent-service.ts) `processQuery`: 工具结果无程序化校验，模型可直接输出或编造参数
- [tools.ts](src/lib/tools.ts) `executeToolCalls`: `generate_report` 接受模型的 `args`，不验证数据来源
- [ai-service.ts](src/lib/ai-service.ts) `buildUserPrompt`: 格式化 commits 时**丢弃了 SHA 标识符**（第197行），报告 LLM 无法自我验证

#### 改进方案：架构级绕过（Architectural Bypass）

核心思路：**不让模型有机会处理/传递提交数据**，在工具结果层捕获真实 commits 后直接程序化生成报告。

**变更 1 — [agent-service.ts](src/lib/agent-service.ts)**：
- 新增 `inferReportOptions(query)` — 从自然语言推断报告选项
- 在 tool-call 循环中**拦截 `generate_report` 调用**：只提取 options，不执行
- **捕获 `get_github_commits` 返回的真实 commits**
- 短路循环，直接调用 `aiService.generateReport(realCommits, options)`
- 无真实数据（空仓库/404/错误）时回退到原有逻辑

**变更 2 — [ai-service.ts](src/lib/ai-service.ts)**：
- 在提交列表中加入 SHA 标识符（如 `` `a1b2c3d` ``），让报告 LLM 可追溯
- 强化 system prompt 中反虚构指令

#### 边界情况处理
| 场景 | 行为 |
|------|------|
| 模型正确调用两个工具 | 绕过触发，用真实 commits + 模型提取的 options |
| 模型只调 get_commits 未调 generate_report | 绕过触发，用 `inferReportOptions(query)` 回退 |
| 空仓库（commits=[]） | 跳过绕过，走模型自身响应 |
| 404/401 错误 | 跳过绕过，走模型自身错误消息 |
| 模型完全不调用工具 | 跳过绕过，回退原逻辑（极罕见） |

#### 验证
```bash
npm run test:eval:real          # 真实评估
npm run test:eval:ai -- --real  # AI Judge 评估（预期通过率 85%+）
```

#### 验证结果（2026-05-03）
| 指标 | Iter 4 (优化前) | Iter 5 (绕过后) | 变化 |
|------|-----------------|----------------|------|
| AI Judge 通过率 | 64.3% (9/14) | **85.7% (12/14)** | **+21.4pp** |
| AI Judge 平均分 | 7.79/10 | **8.30/10** | **+0.51** |
| 失败原因 | 3/5 因虚构数据 | 0/2 因虚构数据 | ✅ 核心问题已解决 |

**关键结论**：架构绕过生效后，所有失败用例均不再由数据虚构引起。剩余 2 个失败为 Agent 自身问题（日期参数推断、超长仓库名解析）。

```
Layer 1: 修复评估系统 (2-3天) ✅
  ├─ 修复 API 调用链路 (0.5天) ✅
  ├─ 实现 LLM-as-Judge (1天) ✅

  └─ 报告升级 (0.5天) ✅

Layer 2: 结构化日志 (2天) ✅
  ├─ Logger 工具类 (0.5天) ✅
  ├─ 关键路径埋点 (0.5天) ✅
  └─ Stats API (1天) ✅

Layer 3: 数据驱动迭代 (3-5天)
  ├─ 基线测量 (0.5天) ✅
  ├─ 第1轮优化 (1天) ✅
  ├─ 第2轮优化 (1天) ✅ ← 消除内容虚构，通过率 64.3% → 85.7%
```

优先级建议：**Layer 1 → Layer 3 → Layer 2 → Layer 4**（先能测、再优化、再看线上、最后防退化）。
